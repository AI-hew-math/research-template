> 5. "검토 받자" 하면 GPT에 복붙할 txt 파일이 자동 생성되게 해줘
> 6. 검토 범위는 마지막 검토 이후 ~ 지금까지

## 4. 판단 근거 (Reasoning)

### 이 접근을 선택한 이유
- GitHub repo로 만든 이유: 새 컴퓨터/폴더에서도 URL 하나로 작동해야 해서
- README에 모든 것을 담은 이유: Claude가 파일 하나만 읽으면 즉시 이해 가능
- dual-source(packet + 전체과정): review packet만으로는 Claude가 자기 합리화 가능. 전체 과정을 함께 줘야 GPT가 교차 검증 가능
- txt 파일 자동 생성: 사용자 수고 최소화 (터미널 긁기 불필요)

### 고려한 대안
| 대안 | 선택하지 않은 이유 |
|------|-------------------|
| ~/.claude/CLAUDE.md에만 규칙 저장 | 새 컴퓨터에서 재설정 필요 |
| GPT API 자동 호출 | API 키 없음, 비용 발생, 수동부터 시작이 합리적 |
| review packet만 제공 | Claude의 자기 합리화 위험 |
| 터미널 복붙만 사용 | GPT가 맥락 파악 어려움, 노이즈 많음 |
| public repo | private으로도 git 인증 있으면 clone 가능 |
| 작업 유형별 미니 체크리스트 | 범용 템플릿의 단순함이 더 중요. 과도한 구조화 |

### 불확실한 부분
- GPT가 이 형식을 받았을 때 실제로 유용한 피드백을 주는지 아직 검증 안 됨
- Part 2(전체 과정)가 매우 길어질 경우 GPT 토큰 한계에 걸릴 수 있음
- Claude가 Part 2를 재구성할 때 무의식적으로 중요한 부분을 빠뜨릴 가능성

1. 전체 시스템 설계(GitHub repo + txt 자동 생성 + dual-source)가 논리적으로 타당한가?
2. README.md가 Claude에게 충분히 명확한 지침서인가? 모호한 부분은?
3. Part 1 + Part 2 구조가 검토에 효과적인가?
4. 이 워크플로우에서 사용자 부담을 더 줄일 수 있는 방법이 있는가?
