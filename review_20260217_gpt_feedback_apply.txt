### 상세
README.md에 추가된 내용:

1) 검토 범위 섹션에 git tag 마킹 규칙:

   - API키, 인증헤더, 비밀번호, SSH정보, .env → ***REDACTED***
   - 원칙: 과잉 마스킹 > 유출

## 4. 판단 근거 (Reasoning)

### 이 접근을 선택한 이유
- git tag: "마지막 검토 이후"를 규칙이 아닌 데이터로 만듦. 간단하고 git 기본 기능
- 마스킹: dual-source 방식에서 터미널 과정을 외부 AI에 전달하므로 유출 방지 필수
- 길이 제한: GPT 토큰 한계 대비. 50 Step/10줄은 실용적 상한선

### 고려한 대안
| 대안 | 선택하지 않은 이유 |
|------|-------------------|
| 별도 CLI 스크립트(cr.py) | Claude Code가 실행 환경이므로 별도 도구 불필요. 유지보수 대상만 늘어남 |
| install 명령어 | URL → clone → 읽기가 이미 설치 과정. 추가 스크립트 불필요 |
| 리뷰 결과 파일 저장/추적 | 현 단계에서 과도. git tag만으로 충분 |
| .cross_review/last_commit.txt 마커 | git tag가 더 깔끔하고 별도 파일 관리 불필요 |

### 불확실한 부분
- 50 Step / 10줄 제한이 실제 사용에서 적절한지는 몇 번 써봐야 알 수 있음
- git tag 방식이 비-git 프로젝트에서는 작동하지 않음 (현재는 문제 없음)

## 5. 검토 요청 (Review Questions)
1. GPT 피드백 5건 중 3건 반영 / 2건 기각 판단이 타당한가?
2. 기각한 "별도 CLI 스크립트" 결정이 맞는가? (Claude Code가 도구 역할을 한다는 논리)
3. git tag 방식이 검토 시점 마킹으로 충분한가?
4. 민감정보 마스킹 목록에 빠진 패턴이 있는가?
